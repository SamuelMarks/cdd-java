# Publishing Generated Client Libraries

This guide outlines how to manage, publish, and automate the lifecycle of the client libraries generated by `cdd-java` from an OpenAPI specification.

## Keeping the Client Library Up-to-Date

When your backend API changes, the client library must be updated to reflect those changes. The best practice is to automate this process using CI/CD.

### Example: GitHub Actions Cronjob

You can set up a GitHub Action to periodically fetch the latest `openapi.json` from your server, run `cdd-java` to regenerate the client library, and automatically open a Pull Request (or directly publish) if changes are detected.

```yaml
name: Update Client Library from OpenAPI

on:
  schedule:
    - cron: '0 0 * * *' # Run daily at midnight
  workflow_dispatch: # Allow manual triggering

jobs:
  update-client:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3

      - name: Fetch latest OpenAPI spec
        run: curl -s https://api.example.com/openapi.json > spec.json

      - name: Setup Java
        uses: actions/setup-java@v3
        with:
          java-version: '11'
          distribution: 'temurin'

      - name: Run cdd-java Code Generation
        run: ./cdd-java from_openapi -i spec.json > src/main/java/com/example/client/ApiClient.java

      - name: Check for changes
        id: git-check
        run: |
          git diff --exit-code || echo "changes=true" >> $GITHUB_OUTPUT

      - name: Create Pull Request
        if: steps.git-check.outputs.changes == 'true'
        uses: peter-evans/create-pull-request@v5
        with:
          commit-message: "feat: update client library from latest OpenAPI spec"
          title: "Automated Client Library Update"
          branch: "auto-update-client"
          body: "This PR contains automatically generated updates to the API client library."
```

## Publishing the Generated Client

Once generated, the client library needs to be built and distributed to consumers.

### Top 5 Build Systems for Generated Clients

1. **Maven:** Best for standalone client libraries. Easy to template a `pom.xml` alongside the generated Java code.
2. **Gradle:** Highly flexible. Great if the generated client is part of a larger multi-project build or requires complex post-processing.
3. **Ant:** Generally avoided for new client libraries, but can be scripted to compile generated `.java` files directly into a `.jar` without a dependency manifest.
4. **SBT:** Useful if the Java client is being consumed primarily by Scala microservices.
5. **Bazel:** Excellent for monorepos where the API server, `cdd-java` generation step, and client library consumption are all defined as inter-dependent Bazel targets.

### Top 5 Locations/Ways for Publishing Generated Packages

1. **GitHub Packages:** The most convenient option if your source code and GitHub Actions are already on GitHub. It tightly couples the generated code repository with its package distribution.
2. **Maven Central:** Ideal for public APIs (e.g., a public Stripe or Twilio client). Requires significant setup (GPG signing, namespace validation), which can be complex to automate cleanly in CI for auto-generated code.
3. **GitLab Package Registry:** The equivalent of GitHub Packages for organizations using GitLab CI. Seamless token integration (`$CI_JOB_TOKEN`) for publishing.
4. **AWS CodeArtifact:** Best for internal corporate APIs. The CI pipeline runs `aws codeartifact get-authorization-token`, injects it into Maven/Gradle, and publishes the client securely for internal microservices to consume.
5. **JFrog Artifactory:** The standard choice for large enterprises. Generated clients are deployed to an internal Artifactory repository, often versioned dynamically based on the OpenAPI version or Git commit hash (e.g., `1.2.0-SNAPSHOT` or `1.2.0-rc.git.abc1234`).
